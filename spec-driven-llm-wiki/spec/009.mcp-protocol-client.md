# Tujuan

Menambahkan dukungan MCP (Model Context Protocol) client ke self-improve-cli sehingga agent loop dapat menemukan dan memanggil tool dari server MCP eksternal. MCP server dikonfigurasi via `.selfimprove/mcp.json`, tool ditemukan secara otomatis saat startup, dan tool call di-bridge ke format OpenAI function-calling yang sudah digunakan agent loop.

Integrasi dilakukan melalui pola `options.tools` + `options.toolHandlers` yang sudah ada (sama seperti mmx-tools.js), tanpa mengubah dispatch core di `executeTool`.

# Kenapa

1. **Ekosistem tool yang luas** — MCP adalah protokol terbuka yang didukung banyak vendor (Anthropic, OpenAI, dll). Server MCP sudah tersedia untuk filesystem, database, browser automation, dan lainnya. Dengan mendukung MCP, self-improve-cli mendapat akses ke ratusan tool tanpa menulis integrasi satu per satu.

2. **Zero dependency** — Implementasi menggunakan plain Node.js built-ins (`child_process`, `http`, `EventEmitter`). Tidak perlu library eksternal. JSON-RPC framing sederhana (header `Content-Length` + JSON body) bisa diimplementasi manual.

3. **Konsisten dengan arsitektur yang ada** — Pola `TOOL_SCHEMAS` + `toolHandlers` sudah terbukti di mmx-tools.js. MCP tools hanya perlu menghasilkan array schema dan map handler yang sesuai, lalu di-merge saat agent loop dijalankan.

4. **Isolasi failure** — Setiap MCP server berjalan di subprocess terpisah (stdio) atau koneksi HTTP terpisah. Crash di satu server tidak mempengaruhi server lain atau agent loop itu sendiri.

5. **Konfigurasi deklaratif** — File `.selfimprove/mcp.json` memisahkan konfigurasi MCP dari kode. User bisa menambah/hapus server tanpa mengubah source code.

## Config Format

File: `.selfimprove/mcp.json`

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@mcp/server-fs", "/path/to/root"],
      "env": { "NODE_OPTIONS": "--max-old-space-size=512" }
    },
    "remote-api": {
      "url": "https://example.com/mcp",
      "headers": {
        "Authorization": "Bearer ${MCP_API_TOKEN}"
      },
      "timeout": 60000
    }
  },
  "defaults": {
    "toolPolicy": "ask",
    "timeout": 30000,
    "reconnect": true,
    "maxReconnectAttempts": 3
  }
}
```

Each entry in `mcpServers` must have **either**:
- `command` (string, required) + `args` (array, optional) + `env` (object, optional) → stdio transport
- `url` (string, required) + `headers` (object, optional) → HTTP transport

Environment variable interpolation in `headers` values: `${VAR_NAME}` is replaced with `process.env[VAR_NAME]` or left as-is if not set.

## Tool Naming Convention

MCP tools are prefixed to avoid collisions:

```
mcp__<serverName>__<toolName>
```

Examples:
- Server `"filesystem"`, tool `"read_file"` → `mcp__filesystem__read_file`
- Server `"github"`, tool `"create_issue"` → `mcp__github__create_issue`

The prefix ensures no collision with built-in tools (`read_file`, `search`, etc.) or mmx tools (`mmx_search`, `mmx_text_chat`).

## Tool Policy

MCP tools follow the same permission system as built-in tools. Default policy for MCP tools:

| Policy Level | Behavior |
|-------------|----------|
| `ask` (default) | Prompt user before each MCP tool call |
| `auto` | Execute without prompting (same as built-in in `auto_approve` mode) |
| `deny` | Block all MCP tool calls from this server |

Policy can be set:
1. Per-server in `mcp.json`: `{ "command": "...", "toolPolicy": "auto" }`
2. Globally in `defaults.toolPolicy`
3. In profile overlay: `tool_policy["mcp__*"]`

## MCP Protocol Lifecycle

Each MCPClient follows this sequence:

```
1. Transport.connect()
   ├── stdio: child_process.spawn(command, args, { stdio: ['pipe','pipe','pipe'], env })
   └── http: establish session (no persistent connection needed for POST-based)

2. sendRequest("initialize", { protocolVersion, capabilities, clientInfo })
   ← response: { protocolVersion, capabilities, serverInfo }

3. sendNotification("notifications/initialized")

4. sendRequest("tools/list", {})
   ← response: { tools: [{ name, description, inputSchema }, ...] }

5. [Ready] sendRequest("tools/call", { name, arguments: {...} })
   ← response: { content: [{ type: "text", text: "..." }, ...], isError?: bool }
```

### Protocol Version

Client advertises `protocolVersion: "2024-11-05"`. If server returns a different version, log a warning but proceed (best-effort compatibility).

### JSON-RPC Framing (stdio)

Messages over stdin/stdout use the standard JSON-RPC over stdio framing:

```
Content-Length: <number>\r\n
\r\n
<json bytes>
```

Each message is a complete JSON-RPC request or response:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": { ... }
}
```

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { ... }
}
```

Notifications (no id, no response expected):

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
```

### HTTP Transport

For remote servers, each request is an HTTP POST:

```
POST <url>
Content-Type: application/json
Authorization: Bearer <token>  (if headers configured)

{ "jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": { ... } }
```

Response is the JSON-RPC response body directly.

SSE (Server-Sent Events) is used for server-initiated notifications if the server supports it. The client opens a GET connection to `<url>/sse` and listens for events. This is optional — if the server doesn't offer SSE, notifications are simply not received.

## Tool Bridge Conversion

### MCP inputSchema → OpenAI parameters

MCP tools return `inputSchema` which is a JSON Schema object. This maps almost directly to OpenAI function-calling `parameters`:

```javascript
// MCP tool from tools/list:
{
  name: "read_files",
  description: "Read multiple files",
  inputSchema: {
    type: "object",
    properties: {
      paths: { type: "array", items: { type: "string" } },
      encoding: { type: "string", default: "utf-8" }
    },
    required: ["paths"]
  }
}

// Converted to OpenAI schema:
{
  type: "function",
  function: {
    name: "mcp__filesystem__read_files",
    description: "[MCP:filesystem] Read multiple files",
    parameters: {
      type: "object",
      properties: {
        paths: { type: "array", items: { type: "string" } },
        encoding: { type: "string" }
      },
      required: ["paths"],
      additionalProperties: false
    }
  }
}
```

Conversion rules:
- `name` → prefixed with `mcp__<server>__`
- `description` → prefixed with `[MCP:<server>] `
- `inputSchema` → used as `parameters` with `additionalProperties: false` added
- Strip `default` from properties (OpenAI doesn't use them in schema)
- If `inputSchema` is missing or invalid, use `{ type: "object", properties: {} }`

### Tool result extraction

```javascript
// MCP tools/call response:
{
  content: [
    { type: "text", text: "file contents here" },
    { type: "image", data: "base64...", mimeType: "image/png" },
    { type: "resource", resource: { uri: "file:///...", text: "..." } }
  ],
  isError: false
}

// Extracted for agent loop:
// → "file contents here"
// (images and resources are ignored — only text content is returned)
// If isError: true → throw Error with text content as message
// If multiple text items → join with "\n"
```

## Agent Loop Integration

In `src/agent.js`, the `runAgentTask` function or the `startChat` function initializes MCP before running:

```javascript
// At the point where mmx-tools are conditionally merged:
let toolsToUse = [...TOOL_SCHEMAS];
const toolHandlers = { ...options.toolHandlers };

if (enableMmx) {
  toolsToUse.push(...MMX_TOOL_SCHEMAS);
  Object.assign(toolHandlers, MMX_TOOL_HANDLERS);
}

// NEW: MCP integration
let mcpManager = null;
const mcpConfig = await loadMcpConfig(root);
if (mcpConfig.mcpServers && Object.keys(mcpConfig.mcpServers).length > 0) {
  mcpManager = new MCPManager(root, mcpConfig);
  const { schemas, handlers } = await mcpManager.discover();
  toolsToUse.push(...schemas);
  Object.assign(toolHandlers, handlers);
}

// Pass to agent loop
const agentOptions = {
  ...options,
  tools: toolsToUse,
  toolHandlers,
  mcpManager  // for shutdown later
};
```

Shutdown after agent loop completes:

```javascript
if (mcpManager) {
  await mcpManager.shutdown();
}
```


`mcp add` options:

- `--args <a,b,c>` — comma-separated args

# Specification

## Module: `src/mcp-transport.js`

### Class: `StdioTransport`

```
constructor(serverConfig)
  - serverConfig.command: string (required)
  - serverConfig.args: string[] (default [])
  - serverConfig.env: object (merged with process.env)
  - serverConfig.timeout: number (default 30000)

async connect()
  - Spawn child: child_process.spawn(command, args, { stdio: ['pipe','pipe','pipe'], env })
  - Start reading stdout with content-length parser
  - Emit 'connected' event
  - If spawn fails, throw Error with details

async send(message)
  - Serialize message to JSON
  - Write Content-Length header + body to child.stdin
  - Return promise that resolves with matching response (id-based)
  - Timeout after serverConfig.timeout ms → reject with timeout error
  - If child exits before response → reject with connection error

async sendNotification(message)
  - Same framing as send(), but no id → no response expected
  - Fire-and-forget

disconnect()
  - Kill child process (SIGTERM, then SIGKILL after 5s)
  - Reject all pending requests
  - Emit 'disconnected' event

Events:
  'connected'    — transport ready
  'disconnected' — transport closed (error or intentional)
  'error'        — (err) child crashed, malformed message, etc.
```

**Content-Length parser** (for reading from child stdout):

```javascript
// Buffer-based state machine:
// State 1: Read until "\r\n\r\n" to get headers
// State 2: Parse "Content-Length: N" from headers
// State 3: Read exactly N bytes as JSON body
// State 4: Emit parsed message, return to State 1
```

### Class: `HTTPTransport`

```
constructor(serverConfig)
  - serverConfig.url: string (required)
  - serverConfig.headers: object (default {})
  - serverConfig.timeout: number (default 30000)
  - Resolve ${ENV_VAR} in header values

async connect()
  - Validate URL is parseable
  - Optionally test connectivity with a simple POST
  - Emit 'connected' event

async send(message)
  - HTTP POST to serverConfig.url
  - Headers: Content-Type: application/json + configured headers
  - Body: JSON.stringify(message)
  - Parse JSON response body
  - Timeout: AbortController + serverConfig.timeout
  - Return parsed response.result or throw on response.error

async sendNotification(message)
  - Same as send() but fire-and-forget (no response expected)
  - POST and ignore response body

disconnect()
  - Abort any in-flight requests
  - Close SSE connection if active
  - Emit 'disconnected' event

Events:
  'connected'    — transport ready
  'disconnected' — transport closed
  'error'        — (err) HTTP errors, parse failures
```

**SSE listener** (optional, for server-initiated notifications):

```javascript
// GET <url>/sse with Accept: text/event-stream
// Parse event stream for JSON-RPC notifications
// Reconnect on connection drop with exponential backoff
// Only started if server capabilities include SSE
```

## Module: `src/mcp-client.js`

### Class: `MCPClient`

```
constructor(serverName, serverConfig, defaults)
  - serverName: string (used in tool prefix)
  - serverConfig: object (from mcpServers[name])
  - defaults: object (from mcp.defaults)
  - Create transport based on config:
    - Has "command" → new StdioTransport(serverConfig)
    - Has "url" → new HTTPTransport(serverConfig)
    - Neither → throw Error

async initialize()
  1. await transport.connect()
  2. Send "initialize" request:
     {
       protocolVersion: "2024-11-05",
       capabilities: {},
       clientInfo: { name: "self-improve-cli", version: "<from package.json>" }
     }
  3. Validate response has serverInfo
  4. Send "notifications/initialized" notification
  5. Return { serverInfo, capabilities }

async listTools()
  - Send "tools/list" request with empty params
  - Return response.tools array
  - Cache result for subsequent calls

async callTool(toolName, arguments)
  - Send "tools/call" request: { name: toolName, arguments }
  - Extract text content from response.content array
  - If response.isError → throw Error(extracted text)
  - Return extracted text (join multiple text items with "\n")

getToolSchemas()
  - Return cached tools converted to OpenAI function-calling format
  - Each tool:
    {
      type: "function",
      function: {
        name: `mcp__${this.serverName}__${tool.name}`,
        description: `[MCP:${this.serverName}] ${tool.description || tool.name}`,
        parameters: convertInputSchema(tool.inputSchema)
      }
    }

getToolHandler(toolName)
  - Return async function: (root, args, options) => {
      return this.callTool(toolName, args);
    }
  - The function is bound to this MCPClient instance

async shutdown()
  - await transport.disconnect()
  - Clear cached tools
```

### Class: `MCPManager`

```
constructor(root, mcpConfig)
  - root: string (workspace root path)
  - mcpConfig: object (loaded from .selfimprove/mcp.json)
  - this.clients = new Map()  // serverName → MCPClient
  - this.schemas = []         // merged OpenAI tool schemas
  - this.handlers = {}        // merged tool handlers

async discover()
  - For each entry in mcpConfig.mcpServers:
    1. Create MCPClient(serverName, serverConfig, mcpConfig.defaults)
    2. Try await client.initialize()
       - On failure: log warning, skip this server, continue
    3. Try await client.listTools()
       - On failure: log warning, skip this server, continue
    4. Store schemas and handlers from client
    5. Add client to this.clients map
  - Return { schemas: this.schemas, handlers: this.handlers }
  - Never throw — individual server failures are logged and skipped

async shutdown()
  - For each client in this.clients:
    - Try await client.shutdown() (don't let one failure stop others)
  - Clear this.clients, this.schemas, this.handlers

getClient(serverName)
  - Return MCPClient or undefined

getToolPolicy(serverName)
  - Check serverConfig.toolPolicy → defaults.toolPolicy → "ask"
```

### Function: `convertInputSchema(inputSchema)`

```javascript
// Converts MCP inputSchema to OpenAI-compatible parameters schema.
// Rules:
// 1. If inputSchema is falsy or not an object, return fallback:
//      { type: "object", properties: {}, additionalProperties: false }
// 2. Clone the schema (shallow copy of top-level + properties)
// 3. Set additionalProperties: false
// 4. Remove "default" fields from property definitions
// 5. Recurse into nested objects/arrays (items.properties)
// 6. Return the cleaned schema
```

### Function: `buildMcpToolBridge(mcpManager)`

```javascript
// Called after mcpManager.discover() completes.
// Returns { mcpToolSchemas, mcpToolHandlers } for merging into agent loop.

// For each client in mcpManager.clients:
//   For each tool in client.getToolSchemas():
//     - Add to mcpToolSchemas
//     - Add to mcpToolHandlers with key = schema.function.name
//       Handler = async (root, args, options) => {
//         // Parse "mcp__<server>__<tool>" to get server and tool name
//         const [, server, tool] = name.split('__');
//         const client = mcpManager.getClient(server);
//         return client.callTool(tool, args);
//       }

// Tool policy enforcement:
//   If policy is "deny" → skip schema (tool not exposed)
//   If policy is "ask" → tool is available but executeTool's ensureAllowed handles prompting
//   If policy is "auto" → tool executes without extra prompting
```

# Error Handling

## Server Crash During Session

```
StdioTransport detects child 'exit' event:
  1. Reject all pending requests with ConnectionError
  2. Emit 'error' event with { type: 'server_exit', code, signal }
  3. MCPClient catches error in callTool()
  4. Returns error message to agent: "[MCP:server] server exited unexpectedly"
  5. Agent can continue using other tools

No automatic reconnect during active agent loop.
Reconnect only happens on next chat session / mcp test.
```

## Timeout on Tool Call

```
Default: 30000ms (30 seconds)
Configurable per-server: { "timeout": 60000 }
Configurable globally: defaults.timeout

Transport.send() uses AbortController:
  - AbortSignal triggers rejection after timeout
  - Error message: "[MCP:server] tool call timed out after ${timeout}ms"
  - Tool call returns error string, agent can continue
```

## Malformed JSON-RPC

```
Content-Length parser encounters invalid frame:
  1. Log warning to stderr: "mcp: malformed message from <server>"
  2. Skip the frame (discard buffered bytes)
  3. Continue reading — don't kill the connection

HTTP response is not valid JSON:
  1. Throw parse error
  2. MCPClient catches, returns error string to agent
```

## Config File Missing

```
loadMcpConfig() returns { mcpServers: {}, defaults: {} }
No servers → no MCPClient instances created
No error thrown, no stderr output
Agent proceeds with built-in tools only
```

## Invalid Config Entry

```
Server entry has neither "command" nor "url":
  1. Log warning: "mcp: server '<name>' has no 'command' or 'url', skipping"
  2. Skip this server
  3. Continue with other servers

Server entry has invalid JSON:
  1. Caught during loadMcpConfig → mcpServers is empty
  2. If parseable but wrong shape, validated in MCPClient constructor
```

```javascript
// tools/list returns: [{ name: "echo", description: "Echo input", inputSchema: {...} }]
// tools/call with name "echo" returns: { content: [{ type: "text", text: JSON.stringify(arguments) }] }
```

# Security Considerations

1. **No arbitrary shell expansion** — `StdioTransport` uses `child_process.spawn(command, args, { shell: false })`. The command must be a direct executable path or name, not a shell expression. Same rule as `run_command` in tools.js.

2. **Environment isolation** — Server `env` is merged with `process.env`, not replacing it. Secrets in env vars are only passed to the server subprocess, not logged.

3. **Header secrets** — HTTP headers may contain tokens. `MCPManager` never logs header values. Config file may contain literal values or `${ENV_VAR}` references. Recommend using env var references for secrets.

4. **Tool policy enforcement** — By default, MCP tools require user confirmation (`ask` policy). This prevents MCP servers from silently executing destructive actions. The `ensureAllowed` function in agent.js handles prompting.

5. **Input validation** — Tool call arguments from the LLM are passed through as-is to the MCP server. No sanitization beyond what JSON allows. The MCP server is responsible for its own input validation.

6. **No tool result size limit** — MCP tool results are passed directly to the agent loop. Very large results may affect context window. Future improvement: truncate large results.
