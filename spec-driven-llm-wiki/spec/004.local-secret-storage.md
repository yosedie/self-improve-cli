# Spec 004: Local Secret Storage for Provider Keys

## Task/Prompt

User does not want API keys through environment variables. They want provider setup to ask for an API key when selecting a provider, store it in a hidden/private local file, and avoid the previous confusion where `api_key_env` was set to the actual key.

## Tujuan

Tambahkan local secret storage yang aman untuk developer machine:

- Store API keys in `.selfimprove/secrets.json`, not config.
- Keep `.selfimprove/` gitignored and hidden by convention.
- Best-effort file permissions: directory `0700`, file `0600`.
- `/connect` prompts for API key after provider selection.
- API key input hidden on TTY.
- `/key` lets user replace current provider key.
- Provider client prefers stored secret, falls back to env var.
- `/config` redacts secrets and only shows `stored_api_key: true/false`.

## Kenapa Tujuan Ini

User wants first-run UX where provider setup just works without manual env vars. Security must avoid printing keys, avoid command-line args for secrets, and avoid storing secrets in normal config.

- Update provider API key lookup to prefer stored secret by `provider_id`.
- Update `/connect` to prompt for key and save it.
- Add `/key` command to replace saved key.
- Update `/config` to show redacted key status.
- Keep environment variable fallback for portability and CI.

## Pseudocode

```text
/connect provider:
  config = connectProvider(provider)
  print provider details
  if TTY:
    key = hiddenPrompt("API key, empty to skip")
    if key:
      saveSecret(provider_id, key)
      chmod .selfimprove 0700 and secrets.json 0600
  print stored_api_key status

provider apiKey:
  stored = getProviderApiKey(root, config.provider_id)
  if stored: return stored
  envKey = process.env[config.api_key_env]
  if envKey: return envKey
  throw Missing API key with instruction to run /connect or /key
```
